**Docker** is a set of products that uses operating system-level virtualization to deliver software in packages called _containers_. Docker automates the deployment of applications within lightweight containers, enabling them to run consistently across different computing environments.

The core software that runs and manages these containers is called **Docker Engine**. Docker was first released in 2013 and continues to be developed by Docker, Inc. The platform includes both free and paid tiers.

Solomon Hykes \[fr\] started the Docker project in France as an internal project within dotCloud, a platform-as-a-service company.[\[5\]](#cite_note-dotcloud-5)

dotCloud Inc. was founded by Kamel Founadi, Hykes, and Sebastien Pahl[\[6\]](#cite_note-Auto38-4-6) during the Y Combinator Summer 2010 startup incubator group and launched in 2011, and renamed to Docker Inc in 2013.[\[7\]](#cite_note-Auto38-5-7)

Docker debuted to the public in Santa Clara at PyCon in 2013.[\[8\]](#cite_note-docker-announcement-8) It was released as open-source in March 2013.[\[9\]](#cite_note-infoq-2013-03-9) At the time, it used LXC as its default execution environment. One year later, with the release of version 0.9, Docker replaced LXC with its own component, _libcontainer_, which was written in the Go programming language.[\[10\]](#cite_note-zdnet-7000030397-10)[\[11\]](#cite_note-infoq-2014-03-11)

In 2017, Docker created the Moby project \[wd\] for open research and development.[\[12\]](#cite_note-moby-12) In March 2026, the Communications of the ACM featured Docker as the cover article in a retrospective article of the past decade.[\[13\]](#cite_note-13)

Docker can use different interfaces to access virtualization features of the Linux kernel.[\[34\]](#cite_note-docker-blog-2014-03-34)

Containers are isolated from one another and bundle their own software, libraries and configuration files; they can communicate with each other through well-defined channels.[\[35\]](#cite_note-Auto38-1-35) Because all of the containers share the services of a single operating system kernel, they use fewer resources than virtual machines.[\[36\]](#cite_note-what-is-a-container-36)

Docker can package an application and its dependencies in a virtual container that can, in principle, run on any Linux, Windows, or macOS computer. This enables the application to run in a variety of locations, such as on-premises, in public _(see decentralized computing, distributed computing, and cloud computing)_ or private cloud.[\[37\]](#cite_note-Linux-37) When running on Linux, Docker uses the resource isolation features of the Linux kernel (such as cgroups and kernel namespaces) and a union-capable file system (such as OverlayFS)[\[38\]](#cite_note-select-storage-driver-38) to allow containers to run within a single Linux instance, avoiding the overhead of starting and maintaining virtual machines. Docker on macOS uses a Linux virtual machine to run the containers.[\[39\]](#cite_note-docker-for-mac-39)

Because Docker containers are lightweight, a single server or virtual machine can run several containers simultaneously.[\[40\]](#cite_note-Chris_K.,_Poweruser_2019-40) A 2018 analysis found that a typical Docker use case involves running eight containers per host, and that a quarter of analyzed organizations run 18 or more per host.[\[41\]](#cite_note-datadog-41) It can also be installed on a single board computer like the Raspberry Pi.[\[42\]](#cite_note-42)

The Linux kernel's support for namespaces mostly[\[43\]](#cite_note-kernel-keyrings-43) isolates an application's view of the operating environment, including process trees, network, user IDs and mounted file systems, while the kernel's cgroups provide resource limiting for memory and CPU.[\[44\]](#cite_note-limit-resources-44) Since version 0.9, Docker includes its own component (called libcontainer) to use virtualization facilities provided directly by the Linux kernel, in addition to using abstracted virtualization interfaces via libvirt, LXC and systemd-nspawn.[\[10\]](#cite_note-zdnet-7000030397-10)[\[34\]](#cite_note-docker-blog-2014-03-34)[\[37\]](#cite_note-Linux-37)[\[45\]](#cite_note-libcontainer-45)

Docker implements a high-level API to provide lightweight containers that run processes in isolation.[\[9\]](#cite_note-infoq-2013-03-9)

The Docker software as a service offering consists of three components:

Software

The Docker _daemon_, called `dockerd`, is a persistent process that manages Docker containers and handles container objects. The daemon listens for requests that are sent via the Docker Engine API.[\[46\]](#cite_note-docker-overview-46)[\[47\]](#cite_note-dockerd-47) The Docker client program, called `docker`, provides a command-line interface (CLI) that allows users to interact with Docker daemons.[\[46\]](#cite_note-docker-overview-46)[\[48\]](#cite_note-cli-usage-48)

Objects

Docker objects are various entities used to assemble an application in Docker. The main classes of Docker objects are images, containers, and services.[\[46\]](#cite_note-docker-overview-46)

-   A Docker _container_ is a standardized, encapsulated environment that runs applications.[\[49\]](#cite_note-digitalocean-49) A container is managed using the Docker API or CLI.[\[46\]](#cite_note-docker-overview-46)
-   A Docker _image_ is a read-only template used to build containers. Images are used to store and ship applications.[\[46\]](#cite_note-docker-overview-46)
-   A Docker _service_ allows containers to be scaled across multiple Docker daemons. The result is known as a _swarm_, a set of cooperating daemons that communicate through the Docker API.[\[46\]](#cite_note-docker-overview-46)

Registries

A Docker registry is a repository for Docker images. Docker clients connect to registries to download ("pull") images for use or upload ("push") images that they have built. Registries can be public or private. The main public registry is Docker Hub. Docker Hub is the default registry where Docker looks for images.[\[46\]](#cite_note-docker-overview-46)[\[50\]](#cite_note-docker-registry-50) Docker registries also allow the creation of notifications based on events.[\[51\]](#cite_note-Auto38-2-51)

A Dockerfile is a text file that commonly specifies several aspects of a Docker container: the Linux distribution, installation commands for the programming language runtime environment and application source code.

An example of a Dockerfile:[\[52\]](#cite_note-52)

ARG CODE\_VERSION\=latest
FROM ubuntu:${CODE\_VERSION}
COPY ./examplefile.txt /examplefile.txt
ENV MY\_ENV\_VARIABLE\="example\_value"
RUN apt-get update
\# Mount a directory from the Docker volume
\# Note: This is usually specified in the 'docker run' command.
VOLUME \["/myvolume"\]
\# Expose a port (22 for SSH)
EXPOSE 22

_Docker Compose_ is a tool for defining and running multi-container Docker applications.[\[53\]](#cite_note-compose-overview-53) It uses YAML files to configure the application's services and performs the creation and start-up process of all the containers with a single command. The `docker compose` CLI utility allows users to run commands on multiple containers at once; for example, building images, scaling containers, running containers that were stopped, and more.[\[54\]](#cite_note-compose-cli-ref-54) Commands related to image manipulation, or user-interactive options, are not relevant in Docker Compose because they address one container.[\[55\]](#cite_note-codeship-55) The `docker-compose.yml` file is used to define an application's services and includes various configuration options. For example, the `build` option defines configuration options such as the Dockerfile path, the `command` option allows one to override default Docker commands, and more.[\[56\]](#cite_note-compose-v3-ref-56) The first public beta version of Docker Compose (version 0.0.1) was released on December 21, 2013.[\[57\]](#cite_note-release-0.0.1-57) The first production-ready version (1.0) was made available on October 16, 2014.[\[58\]](#cite_note-release-1.0.0-58)

_Docker Swarm_ provides native clustering functionality for Docker containers, which turns a group of Docker engines into a single virtual Docker engine.[\[59\]](#cite_note-container-orchestration-tools-59) In Docker 1.12 and higher, Swarm mode is integrated with Docker Engine.[\[60\]](#cite_note-docker-swarm-60) The `docker swarm` CLI[\[61\]](#cite_note-Auto38-3-61) utility allows users to run Swarm containers, create discovery tokens, list nodes in the cluster, and more.[\[62\]](#cite_note-swarm-cli-ref-62) The `docker node` CLI utility allows users to run various commands to manage nodes in a swarm, for example, listing the nodes in a swarm, updating nodes, and removing nodes from the swarm.[\[63\]](#cite_note-docker-node-63) Docker manages swarms using the Raft consensus algorithm. According to Raft, for an update to be performed, the majority of Swarm nodes need to agree on the update.[\[64\]](#cite_note-aquasec-64)[\[65\]](#cite_note-raft-65) In addition to the `docker swarm` CLI, `docker stack` is a tool designed to manage Swarm services with greater flexibility. It can use a configuration file very similar to a `docker-compose.yml`, with a few nuances. Using `docker stack` instead of `docker compose` offers several advantages, such as the ability to manage a Swarm cluster across multiple machines or the capability to work with `docker secret` combined with `docker context`, a feature that allows executing Docker commands on a remote host, enabling remote container management.

_Docker Volume_ facilitates the independent persistence of data, allowing data to remain even after the container is deleted or re-created.[\[66\]](#cite_note-myunidockerguide2-66)
